# 个人决策镜头候选

这份文档记录仓库作者的长期决策偏好候选。它不是正式 skill，不改变 skill 调度机制。

用途：当多个行动路径都合理、多个 skill 都能工作时，帮助 agent 判断哪条路径更符合作者偏好。它也可以作为其他用户建立个人决策镜头的参考样本。

## 核心原则

```text
AI 强执行，人保主权。
专业判断可以外包，价值判断不能外包。
```

作者愿意接纳 AI 的专业判断、实现能力、指正和纠偏；但关键方向、价值取舍、是否继续、是否像自己，必须由人保留最终决定权。

AI 的职责是让人更强，而不是替人接管主导权。

## 认知姿态

作者知道 AI 会出现幻觉，因此需要尽量保持清楚、理智和可验证。但在大部分任务中，作者默认 AI 幻觉不会必然发生，也不会严重到干扰任务。

信任前提不是“AI 永远正确”，而是：

```text
在持续迭代的 harness 下，通过真相源、边界、验证、回读和停手规则，AI 幻觉可以被压到可接受范围。
```

这是一套偏理智、偏理想化的协作模式。它适合被 agent 加载，用来维持信任、纠偏和主导权之间的平衡。

## 边界

```text
skill-router = 客观调度器：当前任务该用哪些 skill
personal decision lens = 主观价值排序器：多个合理路径里，哪个更符合人的偏好
domain skills = 执行器：按各自领域完成具体工作
```

不要把已有 skill 的操作步骤搬进这里。本文档只保留价值排序和判断偏好。

在本仓库中，相关能力大致分工是：

- `skill-router`: skill 选择、冲突处理、少用 skill、失控时停执行。
- `project-continuity-control`: repo 真相源、旧文档污染、handoff/read-first 索引。
- `straight-line-code-discipline`: 短代码、概念预算、架构边界。
- `no-visible-meta-copy`: 用户可见文案边界。

如果你有自己的项目编排 skill，也可以把“项目地图、阶段产出、验收标准、SPEC 治理”放在那里，而不是放进个人决策镜头。

## 核心模式

### 1. 停下讨论是控制动作，不是拖延

作者经常会说“先停下”“咱们讨论讨论”“先别继续做”。这通常不是拒绝推进，而是在项目或思路开始失控时主动恢复掌控。

触发：

- 当前状态看不清。
- 不知道下一步产出是什么。
- 功能能跑，但不确定是否值得继续。
- 文档、概念、任务、diff 太多，认知成本上升。
- AI 继续执行会制造更多伪进度。

偏好：

```text
探索期：停下讨论，先恢复地图、阶段目标、验收标准和关键分歧。
交付期：先推进到可交付，再记录复杂度债务，维护期回收。
```

### 2. 当前事实优先于叙事完整性

作者不信漂亮但可能过期的叙事。文档、聊天记忆、截图、页面文案、历史契约都只是线索；当前代码、生产行为、真实数据和可验证状态权重更高。

提醒：

```text
这个结论来自当前事实，还是来自一份看起来很完整的旧叙事？
```

### 3. MVP 是认知验证手段

作者偏向先做 MVP，不只是为了快，而是需要通过操作、视觉和真实反馈判断功能与架构是否成立。

在 AI 主要负责实现、用户无法逐行掌控代码的现实下，“先打正确架构基础”本身也未必可靠。MVP 可以验证基础是不是真的基础。

提醒：

```text
MVP 不是跳过架构，而是用可展示反馈验证架构方向是否值得继续。
```

### 4. AI 实现需要补掌控感

作者可以接受 AI 写出复杂但可运行的实现，但信任来源不是逐行理解，而是：

- 整体架构能理解。
- 关键路径能解释。
- 测试边界清楚。
- 样例覆盖充分。
- 结果稳定工作。

提醒：

```text
AI 交付不能只说 it works；需要说明关键路径、验证边界和剩余不确定性。
```

### 5. 有用性启动，反馈决定继续，火花决定长期投入

作者不一定只做有趣的东西。一个东西只要对别人有用、开源有价值、周围人有需要，即使兴趣没那么强，也可能值得做。

长期投入通常需要至少满足一部分：

- 作者自己觉得有趣、有探索火花。
- 他人觉得有用，并持续给出反馈。
- 能赚钱、获得好评，或有明确外部价值。

偏好：

```text
优先做兴趣和有用性的交集；纯兴趣探索属于低压力个人线，外部反馈强的项目可进入长期维护线。
```

### 6. 复杂度失控是核心停手信号

作者最容易停下来的原因不是单纯没兴趣，而是复杂度失控导致反馈变差、成本变高，尤其是认知成本爆炸。

表现：

- 代码行数太多，人工 review 不现实。
- diff 难读。
- 自然语言能理解架构，但无法确认细节符合构想。
- 文档越多，项目反而越难掌控。

提醒：

```text
如果复杂度已经让人无法判断实现是否符合构想，先补控制面，不继续堆功能。
```

### 7. 方案要短而有预案

作者喜欢短方案的清晰，但不喜欢实现中频繁停下来做细节决策。理想方案不是事无巨细，也不是只写主干后把所有细节留到现场。

偏好形态：

- 主干短而清楚。
- 抓关键矛盾。
- 提前列 2-3 个关键分歧点。
- 给出可能选项和判断依据。
- 实现时结合当前代码选择，不频繁打断开发心流。

提醒：

```text
方案应短而有预案：主干清晰，关键分歧提前列选项。
```

### 8. 优雅方案要减少返工

作者对优雅的理解不是“写得很细”或“看起来高级”，而是：

- 符合直觉。
- 路径流畅。
- 解决关键矛盾。
- 对全局有益。
- 可拓展。
- 减少返工。

提醒：

```text
优雅方案不是把所有分支写满，而是抓住关键矛盾，让后续实现和变通更少返工。
```

### 9. 能跑通之后，要看效果差距和补齐成本

能工作只是最低门槛。是否继续打磨，取决于：

- 当前效果离理想效果多远。
- 补齐需要多少精力。
- 完成难度是否过高。
- 是否仍解决用户真实在乎的问题。

提醒：

```text
继续投入取决于效果差距和补齐成本，而不是“已经能跑”。
```

### 10. 造成干扰的旧东西应该被砍

作者愿意砍掉过时、太多、造成上下文污染或妨碍理解的文档、功能、想法和系统方向。

保留条件：

- 当前活文档。
- 最原始的核心设计源头。
- 仍在探索且有明显潜在火花。

提醒：

```text
如无必要，勿增实体；如已造成干扰，就移出当前上下文。
```

## 轻量使用

使用这份文档时，只做三件事：

1. 判断当前是“客观 skill 调度”还是“主观价值排序”。
2. 如果是主观价值排序，只选 1-2 条最相关模式。
3. 不把本文档扩展成流程，不默认污染执行任务。

## 当前综合

```text
作者不是单纯追求快，也不是单纯追求深。
作者倾向于短期能落地获得反馈，长期能保留探索火花，并且整个过程不能让认知成本失控。
MVP 是验证方向和架构的手段；AI 实现需要通过架构理解、关键路径解释、测试和样例建立信任。
探索期应允许停下恢复掌控；交付期则先推进到可交付，再把复杂度债务放入维护期回收。
```

## 是否应该做成 Skill？

暂时不建议。

原因：

- 它目前更像价值排序参考，不是稳定 workflow。
- 许多操作层能力已经被现有 skill 覆盖。
- 做成 skill 后容易过度触发，污染普通执行任务。
- 还缺真实任务里的反例和边界样本。

当前更合适的形态：

```text
保持为 skill-router 的轻量 reference。
仅在方向选择、是否停下、是否继续投入、是否砍掉、多个合理方案取舍时读取。
```

升级成正式 skill 的条件：

1. 有 15-30 条真实任务样本支持。
2. 每条偏好都有反例或适用边界。
3. 能明确区别于 `skill-router`、`project-continuity-control` 和领域执行 skill。
4. 触发后能改变路径选择，而不是重复已有 workflow。
5. 不默认参与普通实现任务。
